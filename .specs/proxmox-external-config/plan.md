# Plan: Proxmox External YAML Config Migration

## Context & Motivation

PR #71 (`feature/multi-instance-proxmox-external`) originally added multi-instance Proxmox support by hard-coding `PROXMOX2_` through `PROXMOX10_` environment-variable blocks in `external-available/proxmox.yml`. The branch was rebased onto `origin/main`, conflict-resolved in `sietch/scripts/traefik_hosts.py`, pushed, and GitHub reports the PR as mergeable. Follow-up review found the approach works but is awkward for users because they must track numeric indexes and because Traefik-template YAML is not valid plain YAML before rendering.

The chosen direction is to replace the indexed env-var approach with an OnRamp-generated config model scoped to Proxmox first: users maintain a simple YAML data file listing Proxmox instances, and OnRamp generates normal Traefik dynamic YAML into `external-enabled/proxmox.yml`. A migration command should convert both current `main`-style single Proxmox env vars and PR #71-style indexed env vars into the new YAML format. `make enable-external proxmox` must become generator-aware so the old copy-based workflow cannot overwrite generated Proxmox config. The user also explicitly requested markdown documentation updates for this feature.

## Constraints

- Platform: Windows host using Git Bash/MINGW64 (`MINGW64_NT-10.0-26200`).
- Shell: `/usr/bin/bash`; prefer Bash for git and project commands.
- Project markers: `Makefile`, `.gitattributes`, `sietch/pyproject.toml`.
- Project command convention: service names are positional arguments, not `NAME=` variables.
- Always use `make` commands for OnRamp service operations; do not use raw `docker compose`.
- Do not edit `*.env` files or commit secrets. Migration must read `services-enabled/.env.external` but not delete or rewrite it.
- Migration must be idempotent: reruns should produce identical YAML/output and must not append duplicates.
- Existing users on `main` may have only `PROXMOX_HOST_NAME` / `PROXMOX_ADDRESS`; users who tried PR #71 may have `PROXMOX2_` through `PROXMOX10_` pairs.
- Generated Traefik config should be normal YAML where practical, avoiding Traefik Go-template loops and yamllint exclusions.
- Scope implementation to Proxmox first while keeping the internal design extensible for future external services. Publicly supported generator/migration service in this PR is `proxmox` only.
- Use a minimal internal service-profile registry shape, initially containing only `proxmox`, so future services can be added without rewriting the generator. Keep it as an in-module dict/dataclass; do not add plugin discovery, a public generic external-service schema, or docs implying non-Proxmox generation support. Unsupported services must fail with a clear message such as `external instance generation is currently supported for: proxmox`.
- Domain handling must be explicit: generated host rules use instance `fqdn` if provided; otherwise `host_name` plus `domain`. During migration/generation, `domain` defaults from `HOST_DOMAIN` in `services-enabled/.env` or `services-enabled/.env.external` and fails with an actionable error if neither `domain` nor `HOST_DOMAIN` is available.
- Ownership must be machine-checkable and file-specific:
  - `services-enabled/proxmox.external.yml` is user-owned source config. If migration creates it, include `managed_by: onramp-migration` metadata so later migration runs can safely update env-derived entries. If it exists without that metadata, migration must refuse to modify it unless an explicit replacement flag is used.
  - `external-enabled/proxmox.yml` is generated active Traefik config. It must include a stable generated-file header such as `# Generated by OnRamp; edit services-enabled/proxmox.external.yml instead`. Generator overwrites only files with that header by default.
  - Plain `--force` may update managed files only. Replacing unmanaged source or active config requires an explicit destructive flag such as `--replace-unmanaged` and must create a backup before replacement.
- Legacy env migration must fail by default on incomplete `PROXMOX*_HOST_NAME`/`PROXMOX*_ADDRESS` pairs, duplicate effective router/service keys, duplicate FQDNs, unsafe keys/hostnames/addresses, or unsupported dotenv syntax. Skipping incomplete pairs requires an explicit opt-in flag if implemented.
- Validation grammar must be explicit: instance keys use `[a-z0-9][a-z0-9-]*`; router/service names derive from normalized instance keys and collisions fail; `host_name`, `domain`, and `fqdn` use DNS-label/FQDN syntax with no wildcards, backticks, control chars, template delimiters, or trailing dots unless explicitly documented; `address` is an IP or DNS name only with no scheme/path, with port fixed to the Proxmox profile default `8006` unless a documented `port` field is added and tested.
- Migration key derivation must be deterministic: primary legacy vars map to key `proxmox`; indexed legacy vars map to `proxmox2`...`proxmox10`. Same key with different values, different keys with same FQDN, or normalized name collisions fail unless explicit replacement semantics are used.
- Python tooling commands must run in the Sietch dependency/import context (`cd sietch && uv run ...`) or through Make wrappers that prove the same context.
- Evidence must be redacted: do not paste `.env.external` contents, full diffs of env files, internal IP inventories, or token-like values into plan/archive/PR output. Use fixture values, file status, hashes, and pass/fail summaries instead.
- Documentation updates are in scope and must be included in markdown files.

## Alternatives Considered

| Approach | Pros | Cons | Verdict |
|----------|------|------|---------|
| Keep PR #71 indexed env vars (`PROXMOX2_`...`PROXMOX10_`) | Already mostly implemented; minimal generator code | Users track numbers; capped and repetitive; templated YAML is invalid before Traefik renders; requires yamllint exception | Rejected: works but does not match desired operator UX |
| Use Traefik/Sprig loop over numbered env vars | Less repeated YAML than hard-coded entries | Still requires user-managed numeric env vars; Traefik template syntax support must be carefully verified; output remains templated YAML | Rejected: improves maintainability but not user-facing model |
| Use a single encoded env var list | No separate YAML file | Awkward escaping/parsing; poor readability; harder migration and troubleshooting | Rejected: worse operator ergonomics |
| User-owned YAML data file plus generated Traefik YAML, Proxmox-only public support with internal service profiles | Best UX; idempotent keyed config; valid generated YAML; aligns with scaffold/generator pattern; leaves a clean path for other services later without changing their happy path now | Requires generator and migration script; larger PR scope than env-only | **Selected** |

Opposite-pattern check: If this were a tiny one-off service with no need for multi-instance support or migration, plain env vars would be correct. Here the goal is repeated Proxmox instances plus DR-friendly configuration, so data-file-plus-generator fits better. Do not generalize public support to other external services in this PR; their existing copy-based happy path should remain unchanged.

## Objective

Replace PR #71's indexed-env Proxmox multi-instance implementation with an idempotent YAML-config workflow:

1. Users manage `services-enabled/proxmox.external.yml` as a keyed instance map.
2. OnRamp generates `external-enabled/proxmox.yml` as normal Traefik dynamic YAML.
3. A migration command converts legacy `.env.external` `PROXMOX*` vars into the new YAML without deleting legacy vars.
4. `make enable-external proxmox` has explicit first-run behavior: if `services-enabled/proxmox.external.yml` exists, it invokes/delegates to the generator; if only legacy Proxmox env vars exist, it must either print an actionable instruction to run `make migrate-external proxmox` or run migration only when an explicit opt-in flag/target is used; if an unmanaged `external-enabled/proxmox.yml` already exists, it must refuse or require backup/replacement rather than copying over it.
5. Docs explain the new config, migration, active source of truth, rollback, and generated-file behavior.
6. Automated tests prove migration idempotency, generation correctness, overwrite safety, domain handling, unsupported-service failure, unchanged behavior for non-Proxmox external services, and compatibility with main/PR #71 env-var shapes.

## CLI Contract

The Python module must expose a stable CLI used by tests and Make wrappers:

```bash
cd sietch && uv run python -m scripts.external_instances generate <service> [--base-dir PATH] [--dry-run] [--force] [--replace-unmanaged]
cd sietch && uv run python -m scripts.external_instances migrate <service> [--base-dir PATH] [--dry-run] [--force] [--replace-unmanaged]
```

- Supported service in this PR: `proxmox` only.
- Default `--base-dir`: repository root as seen by Sietch/Make; tests must use temp fixtures.
- Generate default input: `services-enabled/proxmox.external.yml`; output: `external-enabled/proxmox.yml`.
- Migrate default input: `services-enabled/.env` and `services-enabled/.env.external`; output: `services-enabled/proxmox.external.yml`.
- `--dry-run`: prints planned action summary and rendered fixture-safe output, exits 0 on valid input, and writes no files.
- `--force`: permits updates to files already marked as OnRamp-managed/generated; it must not replace unmanaged files.
- `--replace-unmanaged`: explicit destructive replacement mode for unmanaged source/active config; it must create a backup first and report the backup path.
- Stdout: human-readable action summaries and dry-run rendered output using fixture-safe values only.
- Stderr: validation errors, unsupported-service messages, and warnings.
- Exit codes: 0 success/no-op; non-zero for unsupported service, validation failure, unmanaged overwrite refusal, missing domain/input, parse error, or backup failure.
